Section 1

The Loop

Everything you do with Claude Code follows the same four-step cycle. Understand this and everything else clicks.

💬
You Describe
Tell Claude what you want to build
⚡
Claude Writes
Code appears — often working first try
👁️
You Review
Open in browser, check it actually works
🔁
You Refine
Describe what to tweak, repeat

The loop repeats until you're happy — and you can exit at any step.

Section 2

How to Write Better Prompts

Toggle between weak and strong versions to see exactly what makes the difference.

1 — Vague vs. Specific
Make me a website.
⚠ Claude has no idea what kind, what content, what style, or what it's for. You'll get a generic placeholder.
Build a single-page personal portfolio website for a product designer. Include sections for: bio, 3 featured projects with thumbnails, and a contact form. Clean minimal style, white background, dark navy text. Output as a single index.html file with inline CSS.
✓ Claude knows the purpose, audience, content, visual direction, and output format. Far less guessing.
2 — No Context vs. With Context
Add a login page.
⚠ Claude doesn't know what stack you're using, whether auth exists, or how the app is structured. It'll invent assumptions.
I already have a Flask app with a users table (id, email, password_hash). Add a /login route that validates credentials, sets a session cookie, and redirects to /dashboard on success. Use the Werkzeug check_password_hash function I already imported. Don't add any new dependencies.
✓ Claude knows what exists, what to use, and what not to touch. No guessing, no breakage.
3 — Big Ask vs. Phased
Build me a full e-commerce store with product listings, a cart, checkout, payments, and an admin dashboard.
⚠ This is 3–4 weeks of engineering work. You'll get a partial, broken mess — or Claude will refuse to start.
Let's build this in phases. Start with Phase 1 only: Phase 1 — Product listings page - Static HTML/CSS grid of 6 product cards - Each card: image placeholder, name, price, "Add to cart" button (no functionality yet) - Mobile responsive, clean design Stop after Phase 1. I'll ask for Phase 2 (cart logic) once this looks right.
✓ Claude delivers one working thing you can actually test. You stay in control of the scope.
4 — No Format vs. With Format
Build a to-do app.
⚠ You might get a React app with 12 files, a Python backend, and a database — when you just wanted a quick HTML file.
Build a to-do app. Output as a single index.html file with inline <style> and <script>. No frameworks, no build steps, no external dependencies. I should be able to open the file directly in a browser.
✓ You get exactly what you can use immediately. No setup, no npm install, no mystery.
5 — No Constraints vs. With Constraints
Add a chart to my dashboard.
⚠ Claude might introduce Chart.js, D3, or Recharts — adding a library you've never set up and breaking your existing code.
Add a bar chart to my dashboard showing monthly revenue (Jan–Jun, hardcoded data for now). Constraints: no new libraries — use only vanilla JavaScript and the Canvas API. Keep it inside the existing dashboard.html file I already have.
✓ Constraints keep Claude inside your world. The result integrates cleanly with what you already built.

Skills to Learn Alongside

Claude Code handles the syntax. These skills help you steer it better — and debug when things go wrong.

💻
Game Changer
Basic Terminal Commands
cd, ls, mkdir — you'll use these every session. Without them, you can't navigate your own files or run Claude Code itself.
Learn Enough Command Line
🌐
Worth Learning
How HTML/CSS Works
Knowing that HTML is structure and CSS is style lets you give Claude precise instructions and understand what it gives back.
Interneting Is Hard
⚙️
Worth Learning
JavaScript Fundamentals
Variables, functions, arrays — enough to understand what Claude writes and catch obvious bugs before they become real problems.
javascript.info
🔐
Game Changer
Git Basics
Save your work before every big change. One git commit means a broken Claude output can be undone in 10 seconds instead of starting over.
Git — The Simple Guide
🔌
Worth Learning
How APIs Work
Understanding requests and JSON means you can connect Claude's code to real data sources — weather, Spotify, your own database.
Postman API Fundamentals
🔍
Game Changer
Chrome DevTools
Right-click → Inspect. The Console tab shows errors in red. Paste that error into Claude and it almost always knows exactly what's wrong.

Nice to Know
Python Basics
Unlocks backend work — data scripts, Flask apps, automations. Worth learning once you've built a few front-end projects.
learnpython.org

Nice to Know

Components, props, state — helpful once you're building apps with real interactivity. Not needed for your first 10 projects.

Section 4

Golden Rules

Six habits that separate people who get stuck from people who ship. Print this and keep it nearby.

1
Start a new chat for every new project. Old context from a different project confuses Claude and wastes your token budget.
2
Build in phases. Ask for Phase 1 only, test it, then ask for Phase 2. Never describe the whole app at once.
3
Paste the exact error message. When something breaks, copy the full red error text from the console and send it verbatim.
4
Ask Claude to explain anything you don't understand. "Explain this code to me like I'm a beginner" is a completely valid prompt.
5
Save working versions before big changes. Use git commit (or just duplicate the file) before asking Claude to refactor or redesign.
6
Token tip: start fresh when things get messy. Copy just the relevant code into a new chat instead of continuing a broken 50-message thread.
