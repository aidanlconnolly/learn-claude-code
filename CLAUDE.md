# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Running the project

Open `index.html` directly in a browser — no server, no build step. The archived previous version is `index_v1_archive.html`. (A preview-server entry `learn-claude-code` → port 5500 also exists in the workspace `.claude/launch.json`.)

## Architecture

Everything lives in a single `index.html` (~2600 lines). The file is divided into three logical sections:

**CSS** — all styles are inline in `<style>`. Two layout modes are styled independently: `#roadmap-view` (the skill path) and `#lesson-view` (the paginated lesson player). The color system uses CSS variables (`--navy`, `--gold`, `--green`, etc.).

**HTML** — two top-level containers: `#roadmap-view` (contains the `.tab-bar`, `#road-hero`, and `#skill-nodes`) and `#lesson-view`. All content is generated by JavaScript — nothing substantive is static HTML.

**JavaScript** — organized into clearly delimited sections (marked with `// ═══...`):

| Section | What it does |
|---|---|
| `SKILLS` array | 8 skills × 3 lessons × 3–4 pages of course content. |
| `PLAYBOOK` array | 13 skills × 3 lessons × 2–5 pages — the "Claude Code Playbook Setup" tab content (`pb-setup`, `pb-skills`, `pb-data`, `pb-ios`, `pb-prompt`, `pb-design`, `pb-workflow`, `pb-security`, `pb-nextjs`, `pb-api`, `pb-ts`, `pb-arch`, `pb-game`). Same shape as `SKILLS`. Covers machine setup, built-in slash commands, a real backend (Turso + Drizzle + auth), the iOS App Store (Expo + EAS), prompting technique, UI/design (Tailwind + shadcn), pro workflow habits (context/git), security/secrets hygiene, Next.js App Router (server vs client, Server Actions, file-based + `[lang]` routing), the Claude API (server-side call, SSE streaming, structured output / LLM-as-judge), TypeScript (reading types, strict mode, the `_`-prefix convention), frontend architecture patterns (the localStorage+custom-hook pattern, pure logic in `lib/`, typed data files), and building a Canvas browser game (game loop, `dt`, `createState()`). |
| State & localStorage | `getProgress()` / `saveProgress()` / `markLessonDone()` — persists `{ skillId: { done: [bool, ...] } }` under key `ccl_journey_v1`. **Lesson counts are dynamic** (not hardcoded to 3): `getLessonsDone(skillId, lessonCount)` and `markLessonDone(skillId, lessonIdx, lessonCount)` take the unit's `lessons.length` and pad/grow the stored `done` array to fit; `renderRoadmap()` derives the nav total via `arr.reduce((s,sk)=>s+sk.lessons.length,0)` and `allDone` via `doneCount === sk.lessons.length`. So a unit may have any number of lessons. Skill IDs in `PLAYBOOK` are prefixed `pb-` to avoid collisions with `SKILLS` IDs. |
| Tab state | `currentTab` (`'journey'` \| `'playbook'`), `curArray` (snapshot of whichever array is active when a lesson opens), `switchTab(tab)`. `renderRoadmap()` reads `getTabArray(currentTab)` to decide which array to render. |
| Roadmap render | `renderRoadmap()` — populates `#road-hero` and `#skill-nodes` based on the active tab. Called on init and after `backToRoadmap()`. |
| Lesson view | `openSkill(idx)` snapshots `curArray`, then `showLessonView()` → `renderLessonView()` → `renderPage()`. All subsequent lesson-nav functions (`nextPage`, `prevPage`, `getCurrentPage`, `renderLessonView`) reference `curArray[curSkill]`, not `SKILLS[curSkill]` directly. |
| Page renderers | `renderInfo()`, `renderTerminal()`, `renderCode()`, `renderQuiz()` — each returns an HTML string injected into `#lesson-content`. |
| Quiz logic | `answerQuiz(optionIdx)` / `nextQuizQuestion()` called from inline `onclick`. Blocks the Next button until the final question is answered correctly. |
| Terminal animation | `animateTerminal(id, lines)` — types prompt lines character by character, appends output lines immediately. ID format: `term-${curSkill}-${curLesson}-${curPage}`. |

## Adding content

**New page to an existing lesson** — add an object to the relevant `lesson.pages` array inside `SKILLS` or `PLAYBOOK`. Page types:
- `{type:'info', heading, body:[], tip?, keyPoint?, breakdown:[], steps:[]}`
- `{type:'terminal', heading, intro, lines:[{prompt:bool, text}], explain?}`
- `{type:'code', heading, lang, code, explain?}`
- `{type:'quiz', heading, questions:[{q, options:[], correct:int, fb}]}`

**New skill in the Learning Journey** — add to `SKILLS`. Each skill needs: `id` (unique string), `icon`, `title`, `badge`, `badgeClass` (`badge-gold` / `badge-blue` / `badge-gray`), `iconBg` (`bg-gold` / `bg-blue` / `bg-gray`), `tagline`, and a `lessons` array (any length — the engine reads `lessons.length`; 3 is the convention but not required). The roadmap renders an alternating left/right layout automatically; no CSS changes needed for additional skills.

**New tab** — add another data array (follow the `PLAYBOOK` shape), add a button to `.tab-bar` in the HTML with `data-tab` and `onclick="switchTab('...')"`, update `getTabArray()` to return the new array, and update `renderRoadmap()` hero text for the new tab.

**Terminal `lines` format** — `{prompt: true, text: 'command'}` renders as `$ command` with typing animation. `{prompt: false, text: 'output'}` renders as grey output text appended immediately.
